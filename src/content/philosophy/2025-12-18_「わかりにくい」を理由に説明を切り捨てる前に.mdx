# 「『わかりにくい』を理由に説明を切り捨てる前に」
――わかりやすさ・理解・正しさ・説明責任の構造分析

---

## はじめに —— 「わかりにくい」という評価への違和感

あるテキストを複数人で読んで議論していたとき、私には明晰で論点も明確に思えた文章に対して、「わかりにくい」という評価が複数人から出された。

そこまでは問題ではない。理解の経験が人によって異なること自体は当然である。

だが、その直後に、「つまり、このテキストは説明が悪い」という評価が、ほとんど自動的に下されたことに強い違和感を覚えた。

そこでは、「わかりにくい」という主体の内的体験が、ほとんど無反省に、対象であるテキストの性質へと帰属されていた。  
あたかも、

1. わかりにくいと感じた  
2. ゆえに、その説明は悪い  

という推論が、疑われることなく成立しているかのようだった。

本稿は、この違和感――**「わかりにくい」という体験は、そのまま説明の欠陥として扱われてよいのか」**という問いを出発点とする。

## 1. 基本的立場

本稿は、次の立場を取る。

```
「わかりやすい／わかりにくい」は、  
対象に内在する性質ではなく、  
対象を認識する主体において生起する内的現象である。
```

ただし、これは主観相対主義を意味しない。**主観的な体験が生起する構造は、客観的に記述できる**。

本稿の目的は、「わかりやすさ」を価値判断や努力論から切り離し、**認識構造の問題として記述すること**にある。

## 2. 理論的背景 —— 現象学的認識論

本稿は、フッサールに代表される現象学的認識論に依拠する。

現象学において、認識とは対象の写像ではなく、**主体における志向的構成**である。

重要なのは以下の点である。

- 認識は常に「何かについての認識」である（志向性）
- 意味・理解・正当性は、主体において成立する現象である
- 「理解した」という感覚（充実）と、「正当である」という把握（証拠性）は一致を保証されない

本稿はこの枠組みを、「説明」「わかりやすさ」「評価」の問題に適用する。

## 3. 共通例

以下のコードと発言を、すべての分析で共通に用いる。

```ts
type User = { id: string; name: string };

function getUserName(users: User[], id: string): string | undefined {
  return users.find(u => u.id === id)?.name;
}
```

説明者の発言：

> 「`find` は O(1) だから、この実装は高速です。」

本稿では、対象を次に固定する。

```
対象：この `getUserName` 実装の**性能評価**
```

## 4. 重要語彙の定義（共通例による精緻化）

本節では、前節で固定した共通例を最後まで保持したまま、本稿で用いる重要語彙を定義する。

抽象的定義だけではなく、この例において「対象とはこれである」「問いとはここで発生しているこれである」という形で、語彙を構造に固定することが目的である。

### 4.1 対象（object）

**対象**とは、主体がいま「何について」認識しているかを規定する志向の焦点である。

共通例において、説明者が次の発言をしたとする。

> 「`find` は O(1) だから、この実装は高速です。」

このとき、対象は次のように固定されている。

> **対象**：  
> `getUserName` 実装の *性能評価*

重要なのは、ここで対象が

- 実装の仕様  
- 挙動の正しさ  
- コードの可読性  

ではない、という点である。

もし説明の途中で、

> 「reduce を使えば簡潔に書けます」

のような発言が挿入されると、対象は  
**性能 → 表現スタイル** へと暗黙に切り替わる。

この切り替えが明示されない場合、理解者は
「いま何についての話なのか」という**対象の問い**を抱えることになる。

### 4.2 意味（meaning）

**意味**とは、対象を「何として」理解しているかという
カテゴリー付けである。

共通例において、核心となるのは `find` の理解である。

- 意味A：`find` は *配列を先頭から走査する線形探索*
- 意味B：`find` は *高速に要素を取得する検索操作*

説明者が意味Bを前提にし、理解者が意味Aを前提にしている場合、
両者は同じ対象（性能）を見ていながら、まったく異なる結論に至る。

このズレは、

> 「`find` をどういう操作として扱っているか？」

という**意味の問い**として現れる。

### 4.3 地平（perspective）

**地平**とは、対象と意味が成立するために暗黙に前提されている
文脈・条件・知識の総体である。

共通例の性能評価には、少なくとも以下の地平が関与する。

- users の要素数はどの程度か
- 実行頻度はどれくらいか
- 要求性能（SLO 等）は存在するか
- 計算量モデルをどこまで重視するか

説明者がこれらを一切明示せずに「高速」と言った場合、
理解者は次の問いを抱く。

> 「どの条件のもとで高速だと言っているのか？」

これが**地平の問い**である。

### 4.4 妥当性（validity）

**妥当性**とは、「なぜそう言えるのか」という理由づけが
構造的に成立していることを指す。

共通例の発言を妥当性の観点から分解すると、次の構造をもつ。

- 前提P：`find` は O(1) である
- 推論R：O(1) の操作は高速である
- 結論C：この実装は高速である

しかし、前提Pは事実として誤っている。
したがって、この説明は**妥当性を欠く**。

ここで重要なのは、**理解者が「なるほど」と感じたかどうかは、妥当性の成立とは無関係**という点である。

### 4.5 問い（question）

**問い**とは、対象・意味・地平・妥当性のいずれかが主体において未確定であることによって生じる、**充実要求の形式**である。

共通例では、次の問いが典型的に発生する。

- 対象の問い：  
  「いま性能の話をしているのか？」
- 意味の問い：  
  「`find` をどういう操作として扱っているのか？」
- 地平の問い：  
  「どの規模・条件で高速だと言っているのか？」
- 妥当性の問い：  
  「なぜ O(1) だと言えるのか？」

問いは「理解者の怠慢」ではない。**構造上、満たされていない項目の指標**である。

### 4.6 充実（fulfillment）

**充実**とは、問いが満たされ、対象が「そうであるもの」として構成された状態である。

例えば説明が次のように修正されたとする。

> 「この実装は配列を線形探索します（O(n)）。  
> ただし users は数十件程度なので、現実的には十分高速です。」

このとき、

- 対象（性能）
- 意味（線形探索）
- 地平（users が小さい）
- 妥当性（計算量の説明）

が揃い、理解者は「ああ、そういう意味で高速なのか」と把握できる。

これが**充実の成立**である。

### 4.7 証拠性（evidence / Evidenz）

**証拠性**とは、その充実が「そうであると把握してよい」と主体において自己呈示される状態である。

共通例で証拠性が成立するのは、例えば次の場合である。

- 計算量の一般知識と一致している
- users の規模が事実として確認できる
- 実測や比較によって裏づけられている

重要なのは、

> 充実は成立しても、証拠性は成立しないことがある

という点である。

「`find` は O(1)」という説明は、理解者に充実を与えうるが、証拠性を欠いたままである。

### 4.8 小括

以上から、次が明確になる。

- 「わかった」という感覚（充実）は、問いの処理結果である
- 「正当である」という把握（証拠性）は、妥当性と地平に依存する
- わかりやすさは、これらとは別の次元に属する

この区別を失ったとき、「わかりやすい」という判断は、説明の正しさや責任を誤って代理してしまう。

また、問いは次の4型に分類できる。

- 対象問い：いま何についての話か
- 意味問い：それを何として理解しているか
- 妥当性問い：なぜそう言えるか
- 地平問い：どんな前提のもとで言っているか

## 5. 「わかりやすい」「わかりにくい」の構造的位置づけ

本章では、「わかりやすい」「わかりにくい」という評価語を、主観的印象や修辞的表現としてではなく、**認識構造上のどの位置を指している語なのか**という観点から整理する。

結論から先に述べれば、本稿の立場では次のようになる。

- 「わかりやすい／わかりにくい」は**充実（fulfillment）の成立しやすさ**に関する評価である
- それは **証拠性（evidence）の有無や強度を直接には含意しない**

この点を順に確認する。

### 5.1 「わかりやすい」は何についての評価か

日常語としての「わかりやすい」は、しばしば次のように使われる。

- 説明が簡潔である
- 直感的である
- すぐ理解できる

しかし、これらはいずれも曖昧であり、
認識論的には次の問いを残す。

```
「何が、どの段階で、どのように成立しているのか」
```

本稿の語彙を用いれば、「わかりやすい」と評価される状態は、
次のように特徴づけられる。

- *対象*が容易に定まる
- *意味*の解釈に迷いが生じにくい
- *地平*が共有されていると感じられる
- *妥当性*に関する問いが発生しにくい

言い換えれば、**問いが少なく、あるいは即座に解消され、充実が低い摩擦で成立する状態**が「わかりやすい」と経験される。

ここで重要なのは、この評価が **充実の成立に関する主観的経験**を指している点である。

### 5.2 「わかりにくい」とは何が起きている状態か

これに対して、「わかりにくい」と評価される状態では、認識過程のどこかで摩擦が生じている。

具体的には、次のいずれか（あるいは複数）が起きている。

- *対象*が定まらない（いま何についての話か分からない）
- *意味*が確定しない（語や操作をどう解釈すればよいか迷う）
- *地平*が共有されていない（前提や条件が見えない）
- *妥当性*が把握できない（なぜそう言えるのか分からない）

これらはいずれも、第4章で定義した**問いの未解消状態**に対応する。

したがって「わかりにくい」とは、**問いが多く、あるいは問いの所在が不明確で、充実に至らない状態**を指す語である。

ここでも重要なのは、「わかりにくい」が **誤りや不正確さを意味しない**という点である。

### 5.3 「わかりやすさ」と証拠性は独立である

直感的には、「わかりやすい説明ほど正しい」という期待が存在する。
しかし、本稿の枠組みでは、これは成立しない。

理由は明確である。

- わかりやすさは **充実の成立しやすさ**に関わる
- 正しさ（認識的正当性）は **証拠性**に関わる

両者は異なる次元の現象であり、一方から他方は導出できない。

共通例に即して言えば、

> 「`find` は O(1) だから高速」

という説明は、問いがほとんど発生せず、充実は容易に成立する。
その意味で「わかりやすい」。

しかし、妥当性の問いを満たしておらず、証拠性は成立していない。

このように、**わかりやすさは、正しさの指標にはならない**という点が、構造的に導かれる。

### 5.4 なぜ「わかりやすさ」が過剰に重視されるのか

それにもかかわらず、
実践の場では「わかりやすさ」が強い評価軸として機能する。

これは偶然ではない。

わかりやすさが指しているのは、**理解に至るまでの心理的・認知的負荷の低さ**であり、これは主体にとって即時的にアクセス可能な体験だからである。

一方、証拠性の評価には、

- 前提の検討
- 推論の追跡
- 代替案との比較

といった追加的作業が必要であり、即時的な体験としては与えられにくい。

その結果、「わかりやすい」という体験が、「正しい」「説明が良い」という評価を代理してしまうという転倒が生じる。

### 5.5 「わかりやすさ」と証拠性の象限モデル

#### わかりやすい X 誤り

> 「`find` は O(1) だから高速」

妥当性の問い・地平の問いが抑圧され、充実は成立する（＝わかりやすい）が証拠性は成立しない。

#### わかりやすい X 正しい

> 「線形探索だが n が小さいので十分速い」

問いが整理され、充実と証拠性が両立する。

#### わかりにくい X 正しい

> 計算量・最適化・文脈を多層的に説明

正しいが、問いが過剰に発生し充実に失敗しやすい。

#### わかりにくい X 誤り

> 「最適化されているので実質 O(1)」

問いが混乱し、誤りが隠蔽される。

### 5.6 本章の位置づけ

本章で示したのは次の点である。

- 「わかりやすい／わかりにくい」は  認識過程の **充実段階**に関する評価語である
- それは **証拠性とは独立**しており、正しさを保証しない
- にもかかわらず、実践ではしばしば混同される

## 6. 説明者責任／理解者責任の切り分け

| 問い型 | 説明者 | 理解者 |
|---|---|---|
| 対象問い | 非常に強 | 弱 |
| 意味問い | 中 | 中 |
| 妥当性問い | 非常に強 | 弱 |
| 地平問い | 中〜強 | 中 |

## 7. わかりやすさを上げる操作（問い操作）

### 問いの明示  
潜在する問いを言語化する。

### 問いの先送り  
問いの存在を認めた上で扱わないと宣言する。

### 問いの削除  
前提を固定し、問いを成立不能にする。

## 8. 「説明責任」の再定義（重要語彙）

### 従来の問題

説明責任は、努力や善意と結びついた道徳語として誤用されてきた。

### 構造的定義

**説明責任**とは、

> 対象・意味・妥当性・地平に関して発生しうる問いを把握し、  
> それらを明示・先送り・削除のいずれで扱うかを、  
> 構造として示す責務である。

説明責任は、  
「すべて説明する義務」でも  
「相手が理解するまで説明する義務」でもない。

### 帰結

- わかりやすいこと ≠ 説明責任を果たしていること  
- 正しいがわかりにくい説明は、説明責任を果たしている場合がある  
- 「わかりにくい」という評価だけで説明を切り捨てることは、構造的に不当である

## 9. おわりに

「わかりにくい」という体験は、否定されるべきものではない。  
しかしそれは、直ちに対象や説明の欠陥を意味しない。

本稿は、その体験を  
**責任論や感情論ではなく、認識構造として扱うための語彙と枠組み**を提示した。

わかりやすさは主観的だ。  
だが、主観的であることは、思考停止の理由にはならない。
